# iOS課金機能


Swift6対応してからの方が良いかも？

## 実装タスク
### 基本設計
- 課金方式
  - 研究所：月額1,000円のサブスク
  - 講座：非消耗型（講座ごと）
- オフライン時
  - 課金の時にオフラインの場合、機能をブロックするだけ
  - 月額課金中に、オフラインで現在の状態を確認できないときは？
- 一般的な処理部分とDomainに分ける

### クライアント実装
- Product取得
- 購入
- トランザクション検証
- 権利付与・剥奪
- 復元
- 返金・取り消し・家族共有の検出
- UI：ペイウォール、購入中インジケータ、エラー表示、復元ボタン

### エッジケース
- 同一 Apple ID で別端末へ引継ぎ（復元）
- 購入キャンセル/保留(Ask to Buy)/支払失敗
- 価格変更保護・Intro/Promo オファー適用可否
- サブスク切替（アップ/ダウングレード）と残 기간の扱い
- 国/店頭差（ストアフロント）での製品不可時の UI

## ストア側セットアップ
1. 製品作成
  - Product ID 命名規約、表示名/説明、スクリーンショット（必要時）
  - 価格/地域、サブスクはグループ・階層・Intro/Promo オファー設定
  - 審査用メモ（サインイン方法、有料機能到達手順）
2. サンドボックス準備
  - Sandbox テスター作成（複数：新規購入用/既存サブスク/有効期限切れ想定）

## テスト
1. Xcode StoreKit テスト（.storekit）
  - プロジェクトに `.storekit` ファイル追加（製品・サブスクグループ・Intro設定を再現）
  - シミュレーション設定：サブスク更新間隔、初回無料、価格上昇確認プロンプト
  - シナリオ作成：購入→更新→失効→復元→返金→再購入

2. 単体/結合テスト（XCTest + StoreKitTest）
  - `SKTestSession`（StoreKitTest）で製品・ストアフロントを起動
  - 購買フロー：成功/キャンセル/失敗
  - 権利反映：`Transaction.updates` を受けて `EntitlementStore` 更新
  - 復元：`AppStore.sync()` 後の権利再評価
  - 時間経過：自動更新・失効をシミュレート
  - オファー：Intro/Promo の適用条件分岐

3. サンドボックス実機テスト
  - クリーン端末（iCloud サインアウト状態→サンドボックスIDでサインイン）
  - ケース：
    - 新規購入 → 機能解放
    - 端末Bで復元
    - 返金処理（App Store Connect から）→ アプリ側で権利喪失反映
    - ネットワーク不安定時の再試行/UI

5. テスト観点チェックリスト
- [ ]  初回起動で製品一覧取得失敗時のリトライ
- [ ]  ペイウォールでの価格/説明が地域通り
- [ ]  二重タップ防止（多重購入防止）
- [ ]  キャンセル/Ask to Buy 表示
- [ ]  返金・取り消しで権利剥奪
- [ ]  家族共有・所有種別（`ownershipType`）の扱い
- [ ]  ログ＆計測（購入開始/成功/失敗/復元）
- [ ]  VoiceOver/多言語 UI
- [ ]  プライバシー：トラッキング未使用なら明示、レシート/ユーザーID の扱い
