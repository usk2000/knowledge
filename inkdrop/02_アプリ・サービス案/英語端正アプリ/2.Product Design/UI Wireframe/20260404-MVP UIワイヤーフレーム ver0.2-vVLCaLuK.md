# MVP UIワイヤーフレーム ver0.2

## 1. 画面一覧

MVPは基本的に **1画面中心** ですが、初回設定のために軽い導線を追加します。

1. **Initial Level Sheet**
2. **Practice Screen**
3. **Settings Screen（最低限）**
4. **Mini Success Transition（Next後の短い演出）**

核は引き続き **Practice Screen** です。

***

# 2. 初回導線

## Initial Level Sheet

初回起動時のみ表示。

### 目的

- ユーザーに「厳しさ」を先に選ばせる
- ただし重くしすぎない

### 表示案

```
Choose your level

Easy      75
Standard  85   ← Default
Strict    95

[ Start Practice ]
```

### UX意図

- 設定より先に、まず世界観を理解させる
- 「何点で次に進みやすいか」を暗黙に決める

***

# 3. Settings Screen（MVP最小）

## 表示項目

- Practice Level
  - Easy / Standard / Strict
- 現在値表示
  - 75 / 85 / 95

### 表示案

```
Settings

Practice Level
( ) Easy      75
(●) Standard  85
( ) Strict    95
```

### UX意図

- 初回で選んだあとも変更可能
- 深い設定は持ち込まない

## State D1: ResultBelowThreshold

条件：

- `score < selectedThreshold`

### 表示

```
Fit Score

82

Advice
Pull your tongue back for /r/.
                    🔊

[ Retry ]

Skip →
```

### ルール

- Primary は Retry
- Skip は副導線
- Advice は改善指示

***

## State D2: ResultAboveThreshold

条件：

- `score >= selectedThreshold`

### 表示

```
Fit Score

88

Nice fit.

You matched /r/ well.
                    🔊

[ Next ]
[ Retry ]
```

または横並び：

```
[ Next ]   [ Retry ]
```

### ルール

- **Primary は Next単独強調**
- Retry は残すが副導線
- Advice は改善指示ではなく **達成フィードバック**
- Skip は出さない

# 6. 達成時文言の切り替え

## threshold未満

改善指示を出す

例

```
Pull your tongue back for /r/.
Keep the vowel shorter.
```

## threshold以上

達成フィードバックに切り替える

例

```
Nice fit.
You matched /r/ well.
Good rhythm on this phrase.
```

### UX意図

- 「まだ直す」モードと「できた」モードを分ける
- 高スコア時に学習者を疲れさせない

***

# 7. Next押下後の短い達成演出

## Mini Success Transition

Next押下後、次Promptへ即切り替える前に、**短い達成演出** を入れる。

### 表示時間

- 300ms〜700ms程度

### 表示例

```
Nice fit ✓
Next phrase...
```

または

```
Great.
Let’s go to the next one.
```

### UX意図

- 「進んだ感」を与える
- ただしテンポは壊さない

### 注意

- 長い演出は禁止
- フル画面遷移にしない

# 1. 初回Level Sheetの扱い

## 結論

**スキップ不可で問題ありません。**  
しかも、**Default が選択済みの状態で Start を置く**のはUX上かなり良いです。

## 理由

初回ユーザーにとっては、

- 何を基準に次へ進むか
- このアプリがやさしいのか厳しいのか

がまだ分かっていません。  
ここを最初に軽く決めてもらうと、**後のスコア体験の違和感が減ります**。

また、最初から未選択状態にすると、

- 1アクション増える
- 「どれを選べばいいか」で止まる

ので、MVPの思想と逆です。

## 推奨UI

```
Choose your level

( ) Easy      75
(●) Standard  85
( ) Strict    95

[ Start Practice ]
```

## さらに良くするなら

ラベルだけだと意味が弱いので、**1行説明**を足すのが良いです。

例：

- Easy 75 — 進みやすい
- Standard 85 — おすすめ
- Strict 95 — 厳しめ

これで迷いがかなり減ります。

***

# 2. Success Transitionの文言

## 結論

**固定ではなく、スコア帯ごとに変える**のは正しいです。  
さらに、**同じ帯で3種類程度ランダム**にするのも良いです。

## 理由

固定文言だと、短期間で機械的に感じやすいです。  
一方で、完全ランダムにしすぎると、UXの一貫性が崩れます。

なので、

- スコア帯でトーンを合わせる
- その中で数種類回す

のがちょうど良いです。

## 例

### しきい値到達直後

- Nice fit ✓
- Good job ✓
- You got it ✓

### 高めに超えた

- Great fit ✓
- Strong match ✓
- That sounded solid ✓

### かなり高い

- Excellent fit ✓
- Very natural ✓
- That was a great match ✓

***

# 3. 高スコア時のRetry

## 結論

**弱める判断は正しい**です。

## 推奨

高スコア時は

```
[ Next ]

Retry
```

または

```
[ Next ]
Try once more
```

くらいで十分です。

## 理由

高スコア時の主目的は「前進」です。  
Retry を通常ボタンのまま残すと、Next の意味が弱くなります。

***

# 4. Settings導線

## 結論

MVPでは **Headerアイコンで良い** です。

## 理由

- 軽い
- 1画面中心の思想を壊さない
- 初回Sheetとの役割分担が明確

## 注意点

ただし、歯車アイコンだけだと意味が弱い場合があります。  
MVPでは次のどちらかが安定です。

- `⚙︎`
- `Level 85` のような小さなラベル付き導線

後者は少し実用的です。  
ユーザーが「今どのレベルか」を忘れにくいからです。

***

# 5. 将来の自動最適化導入

## 結論

その導入方針は自然です。

- MVP：初回Sheetで選ぶ
- MVP後：自動最適化ON/OFFをSettingsへ
- OFF時：手動レベル選択

これはかなりきれいです。

## UX上の意味

最初は

- ユーザーにコントロール感を渡す

後から

- システムに最適化を任せられるようにする

という流れになります。

これは学習プロダクトとして筋が良いです。
